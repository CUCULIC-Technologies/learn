The Handshake at a Glance

Step by Step

1
User → Browser

The user clicks "Connect iManage"

Tim opens Fathom, sees a banner saying his iManage account isn't linked, and clicks the Connect button. Nothing magical here — it's a plain hyperlink to a Fathom endpoint.

HTTP Request
GET /imanage/connect?returnUrl=/Projects/abc-123
2
Browser → Fathom

Fathom prepares the handshake

The ImanageAuthController.Connect action receives the request. It needs two pieces of cryptographic prep before sending the user to iManage: a PKCE code verifier and a CSRF state.

What Fathom generates
codeVerifier = 43-128 char random string (URL-safe base64)
codeChallenge = base64url( SHA-256( codeVerifier ) )
state = 32 bytes of randomness, hex-encoded
// All three are bound to the user's session via
// an encrypted HttpOnly cookie (IDataProtector).
Why the state? Prevents a malicious third party from initiating a login flow that ends with their code being exchanged in your session. We check state matches on the callback.
3
Fathom → Browser

Redirect the browser to iManage

Fathom doesn't talk to iManage yet — it just hands the browser a URL and a 302. The browser will go ask iManage on its own. Crucially, we send the SHA-256 challenge, not the verifier itself.

HTTP Response
302 Found
Location: https://cloudimanage.com/auth/oauth2/authorize?
  response_type=code&
  client_id=B7723432-F804-…&
  redirect_uri=https://app.fathom/imanage/connect/callback&
  code_challenge=Bm83…lqA&
  code_challenge_method=S256&
  state=7f1a…&
  scope=user
Set-Cookie: fathom.imanage.connect=<encrypted blob> // holds verifier+state
4
Browser → iManage

The browser visits iManage

The user's browser follows the 302 and ends up at cloudimanage.com. iManage receives the authorize request and recognizes Fathom by its client_id. It checks the redirect_uri against its registered list — if it's not on the list, the flow halts here.

What if my redirect_uri isn't registered? iManage's authorize endpoint returns an error page. This is the OAuth spec — a defense against attackers tricking the IdP into sending a code to an attacker-controlled URL. That's why we have to register each environment's callback URL in iManage Control Center.
5
iManage ↔ User

The user proves who they are

iManage shows its own login page. The user types their username and password, possibly does SSO via SAML, possibly does MFA. Fathom never sees any of this. That's the whole point of the OAuth dance — the user's credentials stay between them and the IdP.

The point of OAuth In the old world (and in Fathom's current service-account flow), Fathom would have the user's password and send it to iManage on their behalf. With OAuth, Fathom never sees it. Compromising Fathom doesn't compromise iManage credentials.
6
iManage → Browser

iManage sends back an "authorization code"

After authentication succeeds, iManage 302s the browser back to the registered callback with a short-lived authorization code (10-minute lifetime, single-use). This code is NOT the access token — it's a temporary handle that must be exchanged.

HTTP Response (from iManage)
302 Found
Location: https://app.fathom/imanage/connect/callback?
  code=Z3Q1bF8…&
  state=7f1a… // the same state Fathom generated
7
Browser → Fathom

Fathom validates the callback

ImanageAuthController.Callback wakes up. It pulls the encrypted cookie, decrypts it, and checks: does the state parameter match what we issued? If not — abort. If yes, we have our codeVerifier ready for the exchange.

Validation
// 1. Decrypt the cookie:
{ codeVerifier, state, returnUrl } = protector.Unprotect(cookie)
// 2. Compare states (constant-time):
if (received.state != stored.state) return Unauthorized()
// 3. Drop the cookie — it's single-use.
8
Fathom → iManage

The PKCE exchange — code for tokens

Fathom makes a server-to-server call to iManage's token endpoint. It sends the authorization code, the original codeVerifier (now finally revealed), and its client_secret. iManage verifies that SHA-256(codeVerifier) == codeChallenge from step 3.

HTTP Request
POST https://cloudimanage.com/auth/oauth2/token
Content-Type: application/x-www-form-urlencoded

grant_type=authorization_code&
code=Z3Q1bF8…&
code_verifier=<raw verifier from step 2>&
client_id=B7723432-…&
client_secret=DA74B241-…&
redirect_uri=https://app.fathom/imanage/connect/callback
This is the PKCE moment An attacker who intercepted the code in step 6 can't redeem it without also having the codeVerifier — which never left Fathom's server. The cryptographic binding makes code interception useless.
9
iManage → Fathom

iManage returns the tokens

If everything checks out, iManage issues a fresh access token (use this on API calls) and a refresh token (use this to get a new access token when the current one expires).

HTTP Response
200 OK
Content-Type: application/json

{
  "access_token": "eyJhbGc…",
  "refresh_token": "r1g2T…",
  "token_type": "Bearer",
  "expires_in": 3600
}
10
Fathom → Fathom

Encrypt and persist 🔒

Fathom encrypts both tokens with IDataProtector (purpose fathom.imanage-tokens.v1) and writes a single row to the UserImanageTokens Azure Table, keyed by the user's Entra OID. The tokens never sit unencrypted on disk.

Storage operation
ImanageTokenStore.UpsertAsync(entraOid, tokens)

entity.PartitionKey = entraOid; // the user
entity.RowKey = "imanage"; // one row per user
entity.AccessTokenProtected = protector.Protect(access);
entity.RefreshTokenProtected = protector.Protect(refresh);
entity.AccessTokenExpiry = utcNow + expires_in;

Finally, Fathom 302s the browser back to wherever the user started (the returnUrl we carried through the cookie). The user sees a "Connected ✓" badge. The handshake is done.

The PKCE trick — in 30 seconds

Without PKCE, an attacker who intercepts the authorization code (e.g., via a malicious browser extension or a logged URL) could exchange it for a token. PKCE binds the code to a secret that only Fathom's server knows, even if everything else in the flow leaks.

Step 2 · Sent over the wire
code_challenge = SHA-256(verifier)
Step 8 · Never sent until now
code_verifier = <original random bytes>

iManage verifies SHA-256(verifier) == challenge. A one-way hash means an attacker who saw the challenge can't reverse it to find the verifier. Even capturing every redirect in the user's browser doesn't help — the verifier never leaves Fathom's server until the final server-to-server token exchange.

What Happens When Tokens Expire?

Access tokens are short-lived (typically one hour). Refresh tokens live much longer (days to weeks). Fathom handles the rotation transparently so the user doesn't notice.

① Detect

Before any iManage API call, Fathom checks AccessTokenExpiry < UtcNow + 60s. The 60-second buffer covers clock skew.

② Refresh

POST to /auth/oauth2/token with grant_type=refresh_token + the stored refresh_token. iManage returns a new access_token (and may rotate the refresh_token).

③ Persist + retry

Re-encrypt and upsert. The original API call retries automatically with the new token. If the refresh itself fails (e.g., user revoked access in iManage), the user is asked to reconnect.

Glossary

Authorization Code
A short-lived, single-use token iManage hands to the browser after login. Useless on its own — must be exchanged for an access token.
PKCE (S256)
"Proof Key for Code Exchange." A random verifier is hashed into a challenge; the challenge goes out, the verifier stays home. Binds the authorization code to the originating session.
State
CSRF token bound to the user's session. We check it on the callback so an attacker can't trick a user's browser into completing someone else's OAuth flow.
Access Token
Bearer token used on iManage API calls (header X-Auth-Token). Lifetime ~1 hour. Treat as a password — never log, never transmit unencrypted.
Refresh Token
Long-lived token used to mint new access tokens silently. Lives in the encrypted DB column. May be rotated on each use.
Entra OID
The user's unique identifier in Azure Entra (formerly Azure AD). Fathom uses this as the partition key for token storage so per-user lookups are constant-time.
IDataProtector
ASP.NET Core's at-rest encryption API. Keys live in %LOCALAPPDATA%\ASP.NET\DataProtection-Keys in dev; in prod they should be persisted to Key Vault or Blob.
Redirect URI
The exact URL iManage will send the auth code to. Must be pre-registered in iManage Control Center per environment — no wildcards, no near-misses.
