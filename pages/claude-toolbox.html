I.
The Reviewers & Refiners
04 plugins
Plugin01
security-guidance
The paranoid friend who reads every line.

Surfaces security concerns in your code — SQL injection paths, leaky secrets, sketchy auth flows, dependencies with bad reputations. The voice in your head that says "wait, is this safe?" — but louder, and with citations.

Use it when You're shipping a feature that touches user data, auth, or any string that came from outside your code.
Plugin02
code-review
A second pair of eyes that never gets tired.

A general-purpose review pass over your diff: style consistency, edge cases you missed, naming that's a little off, the function that's grown three heads since last Tuesday. Less "find the bug," more "would I be proud of this in six months."

Use it when You're about to push a branch and want a thoughtful read before a human spends their time on it.
Plugin03
commit-commands
Git plumbing, with manners.

Generates clean commit messages, stages thoughtfully, splits sprawling changes into reviewable commits with intent. The kind of git hygiene you mean to do, then forget at 5:47 PM.

Use it when Your working tree has eleven unrelated changes and you don't want a single "stuff" commit.
Plugin04
code-simplifier
Pruning shears for clever code.

Takes that ternary-inside-a-reduce-inside-a-map and asks, gently, whether maybe a small for-loop and a named variable would do. Optimizes for the next person reading it — who is almost always future-you.

Use it when You wrote something that worked, then re-read it and went "huh."
II.
The Builders
05 plugins
Plugin05
hookify
Extracts a hook from the wreckage.

Turns sprawling component logic into clean, reusable React hooks. State, effects, derived values — pulled out, named well, and given a proper home. Your components get thinner; your codebase gets smarter.

Use it when You've copy-pasted the same useEffect twice and the third one is on the horizon.
Plugin06
feature-dev
From ticket to working code, with intent.

A workflow for building a feature end-to-end: understanding the requirement, sketching the approach, writing the code, wiring up the tests. It plans before it types, which sounds obvious until you've seen the alternative.

Use it when You've got a feature spec and want a methodical pass rather than a vibes-based one.
Plugin07
frontend-design
The aesthetic conscience.

Distinctive, production-grade UI — not generic AI slop. Picks typography with intent, commits to a visual direction, and treats whitespace like a design element instead of an accident. Pairs nicely with a design system.

Use it when You need a landing page, dashboard, or component that doesn't look like every other landing page.
Plugin08
pr-review-toolkit
For when you're the reviewer.

A structured set of moves for reviewing someone else's pull request: read the description, scan the diff with context, ask the right questions, leave comments that improve the code rather than the reviewer's ego.

Use it when You've been assigned three PRs and want to give them more than a thumbs-up.
Plugin09
skill-creator
A meta-tool: builds new tools.

Helps you author your own Claude skills — those reusable SKILL.md files that teach Claude how to do something specific the way your team does it. The path from "we always do X" to "Claude now does X correctly."

Use it when You've explained the same convention to Claude three times and want it written down.

The Extenders
04 plugins
Plugin10
plugin-dev
Workbench for plugin authors.

Scaffolds, structures, and tests Claude Code plugins. Handles the boilerplate of plugin shape so you can focus on the actual capability you're trying to expose. Plugins building plugins, all the way down.

Use it when You want to package a workflow your team uses into something portable and shareable.
Plugin11
agent-sdk-dev
For when Claude needs to be the operator.

Develop against the Claude Agent SDK — building autonomous agents that plan, call tools, and run multi-step workflows. The difference between "Claude answers a question" and "Claude finishes a job."

Use it when You're building an internal agent that needs to do work, not just respond.
Plugin12
mcp-server-dev
Wire Claude into your stack.

Builds Model Context Protocol servers — the bridge between Claude and your databases, APIs, internal tools, and that one weird legacy SOAP endpoint. If you've ever thought "Claude should be able to read this," this is how.

Use it when Your team's tooling lives behind a wall Claude can't currently reach over.
Plugin13
claude-code-setup
The opening ritual.

Bootstraps Claude Code in a new project or environment: installs, configures, points everything at the right places. The setup that should take five minutes and historically takes forty.

Use it when You're starting fresh on a new machine, repo, or onboarding a teammate.
IV.
The Workflows & Curiosities
05 plugins
Plugin14
claude-md-management
The project memory keeper.

Helps you write, maintain, and prune your project's CLAUDE.md — the file that tells Claude how this codebase works, what to never do, and where the bodies are buried. A living document, not a stale comment block.

Use it when Your CLAUDE.md is either empty or has grown into a 600-line novel nobody updates.
Plugin15
playground
A sandbox with a roof.

A scratch space for trying things — quick experiments, throwaway scripts, "what if I just..." moments that shouldn't pollute your real working tree. Sometimes the best engineering happens outside the engineering.

Use it when You want to test an idea before committing to it (literally).
Plugin16
ralph-loop
Iterate until it's right.

A persistent loop pattern: run, observe, adjust, run again. For tasks that need many small passes — fixing flaky tests, tuning a heuristic, narrowing in on a bug whose footprint keeps changing. Named, presumably, after a Ralph.

Use it when The fix is "keep trying things until the failures stop."
Plugin17
explanatory-output-style
Show the working.

An output mode that narrates reasoning alongside results — why this approach, what alternatives were considered, where the tricky bits are. Good for code reviews you're learning from, not just rubber-stamping.

Use it when You want to understand the answer, not just receive it.
Plugin18
learning-output-style
A teacher in the terminal.

Goes a step beyond explanatory — actively tries to teach the concept, with examples, analogies, and gentle quizzing. The output style for picking up a new language, framework, or paradigm rather than just shipping in one.

Use it when You're deliberately learning something — say, picking up a new stack for an internal tool.

The plugin is the container; what's inside is the capability.

A plugin is the installable bundle listed in your claude.json. Each one packages something Claude can use — a skill (a SKILL.md teaching Claude how to do something a particular way), a command (a slash-command you invoke directly), an output style (how Claude communicates while working), an MCP server (a bridge to external systems), or a workflow harness.

Grouping by container type makes the architecture honest: it's the difference between "what does this do for me" and "what does this actually ship."
