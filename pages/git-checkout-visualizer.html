FIRST: THE THREE TREES

// before we can talk about switching, you need to know what's in play

Git doesn't have one "current state" — it has three. Switching branches touches all of them in specific ways. Meet the cast:

[ HEAD ]
A pointer to the commit you're "on." Stored in .git/HEAD.
CHANGES
[ INDEX ]
aka the "staging area." A snapshot of what your next commit will look like. Lives in .git/index.
CHANGES
[ WORKING DIR ]
The actual files on disk that you can see, edit, build, and run.
CHANGES
KEY INSIGHT

When you switch branches, all three trees get updated to match the target branch. The HEAD pointer moves, the index is rewritten, and the files in your folder are physically replaced.

1

VALIDATION & SAFETY CHECK

// git asks: can I even do this without destroying your work?

You type git switch feature-x (or the older git checkout feature-x). Before doing anything destructive, git scans your working directory and index for changes.

→ Resolve "feature-x" to a commit SHA (e.g., a3f9c21...) → Diff your working dir against current HEAD → Diff target commit against current HEAD → Would switching overwrite uncommitted changes?
⚠ DIRTY STATE

If you have uncommitted changes to a file that differs between branches, git will refuse and yell at you: error: Your local changes would be overwritten. Your options: commit, stash, or discard. Git won't silently nuke your work.

If your changes are to files that don't differ between branches, git happily carries them along.

2

MOVE THE HEAD POINTER

// the smallest, fastest part of the whole operation

Git rewrites a single file: .git/HEAD. That's it. One line of text.

$ cat .git/HEAD ref: refs/heads/main
$ cat .git/HEAD ref: refs/heads/feature-x

That's the entirety of "switching branches" from git's bookkeeping perspective. The rest of this guide is about syncing the index and your files to match where HEAD now points.

3

REWRITE THE INDEX

// the staging area becomes a snapshot of the new branch's tip

Git reads the tree object pointed to by the new commit and writes it into .git/index — a binary file listing every tracked file in the repo with its mode, hash, and metadata.

┌─────────────────────────┐ ┌─────────────────────────┐ │ .git/index (before) │ → │ .git/index (after) │ │ ─────────────────── │ │ ─────────────────── │ │ src/App.cs @ab12... │ │ src/App.cs @ff09... │ │ README.md @cd34... │ │ README.md @cd34... │ │ config.json @ef56... │ │ config.json @ef56... │ │ │ │ src/NewFeat.cs @99aa.. │ └─────────────────────────┘ └─────────────────────────┘ (matches main) (matches feature-x)
4

UPDATE THE FILES ON DISK

// the part you actually notice

Git walks the difference between the old tree and the new tree, then physically rewrites your working directory. This isn't magic — it's literal file I/O happening on your SSD.

For each file that differs between branches: → Files only on old branch ........ DELETED → Files only on new branch ........ CREATED → Files with different content ...... REWRITTEN → Identical files ......................... UNTOUCHED

This is why your bin/ and obj/ folders survive a branch switch (they're gitignored), but your Controllers/FathomController.cs can suddenly have totally different contents the moment the command returns.

WHAT DOESN'T CHANGE

// the surprisingly long list of things git leaves alone

The illusion is that you "teleport" to another branch. The reality is far more mundane:

✓ Files in your .gitignore (build artifacts, node_modules, .env) ✓ Untracked files git has never seen ✓ Files identical between the two branches ✓ Your editor's open tabs, breakpoints, terminals ✓ Your shell's environment variables and cwd ✓ Running processes, dev servers, debuggers
⚡ THE GOTCHA

Your dev server is still running with the old code in memory. Your IDE has the old class definitions cached. The files on disk are new — but everything derived from those files (compiled assemblies, hot-reload state, IntelliSense indexes) is stale until rebuilt.

∎

SO. WHAT IS A BRANCH SWITCH, REALLY?

// putting it all together
TL;DR

Git rewrites .git/HEAD to point at a new branch. Then it rewrites .git/index to match that branch's commit. Then it walks your working directory and creates, deletes, and overwrites files until they match too. The whole thing usually finishes in milliseconds, and your editor often catches up before you do.

Now — remember that worktree thing from earlier? All of the above happens in your one working directory. A worktree gives you a second set of "HEAD + index + working directory" backed by the same .git/objects/ database. That's why you can have two branches checked out at once: two HEADs, two indexes, two folders, one shared object store.
